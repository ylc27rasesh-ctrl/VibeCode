The foundations of vibe coding

What vibe coding actually is, the mental model of an app, the universal vocabulary you need before any tool makes sense.

CHAPTER 1.1

What is vibe coding, really?

Vibe coding is a term coined in February 2025 by Andrej Karpathy (co-founder of OpenAI, former AI lead at Tesla). The idea: instead of typing every line of code yourself, you describe what you want in plain English, and an AI model writes the code for you. You read it, run it, see what breaks, and tell the AI to fix or change things. You "surrender to the vibes" of the model.

A year later, in 2026, this is no longer a niche thing. It's the dominant way solo builders, small teams, and even working professionals create software. The tools have gotten genuinely good. Real apps with real users get shipped this way every day.

Two big categories of vibe coding tools, and the difference matters:

Category A
App Builders

You chat. They build the whole thing — frontend, backend, database, deployment. You may never see the code.

Lovable · Bolt · Replit · v0 · Hostinger Horizons · Base44
Category B
AI Coding Agents

They live on your computer or terminal. They write and edit real code files, run commands, often work inside your editor.

Claude Code · Codex · Cursor · Antigravity · Windsurf · Cowork

Category A is where you'll start. Category B is where you'll graduate to when you want more control. By the end of this guide you'll be comfortable in both.

Why this guide exists

Most "vibe coding" content assumes you already know what a terminal is, what GitHub does, or what "deploying" means. This one doesn't. We start from zero and don't skip steps.

CHAPTER 1.2

The mental model of an app

Before any tools, you need a picture of what an app actually is. Almost every modern app has the same three pieces. Once you see them, every tool in this guide will make sense.

Frontend
What the user sees and clicks. Buttons, screens, colors, layout.
↔
Backend
The brain. Handles logic, logins, payments, calling AI.
↔
Database
Where data lives — user accounts, posts, scores, settings.

Plus two helpers:

  • Hosting — the computer on the internet that runs your app 24/7 so people can visit it.
  • Domain name — the friendly address like mycoolapp.com that points to your hosting.

A "full-stack" app means all three layers exist. An AI agent is usually just a backend that talks to an AI model and other services. A simple game can sometimes be just a frontend.

The good news: modern vibe coding tools handle all of this for you when you're starting out. Lovable, Bolt, Replit will spin up frontend + backend + database + hosting from one prompt. Your job is to describe the thing.

CHAPTER 1.3

The plain-English glossary

Bookmark this. Every term here shows up across the rest of the guide and across every tool you'll touch.

Code
Instructions for a computer in a programming language. Looks like const greeting = "hello". You won't write much; AI will.
File & folder
A project is just a folder of text files. index.html, app.js, style.css. The folder is sometimes called a "repo" (repository).
Markdown
A simple way to format text using symbols. **bold** becomes bold. # Heading becomes a heading. Files end in .md. You'll see Markdown everywhere — README files on GitHub, AI agent instructions, Antigravity's "artifacts." Easier to read than HTML.
Terminal / Command Line
A black window where you type text commands instead of clicking. On Windows it's PowerShell or Command Prompt. Vibe coding tools have made it mostly optional.
VS Code
Microsoft's free code editor. A souped-up Notepad — a window for editing files. Most popular code editor on the planet. Many AI tools are built on top of it.
IDE
Integrated Development Environment — a code editor with extras built in (terminal, file browser, debugger). VS Code, Cursor, Antigravity, Windsurf are all IDEs.
Node.js
Lets your computer run JavaScript outside a browser. Many AI tools need it installed. You install it once and forget about it.
npm / npx
Comes with Node.js. npm ("Node Package Manager") installs little JavaScript programs called "packages." npx runs a package without installing it permanently. You'll type npx antigravity-ide to launch Antigravity, for example.
Git
Tracks every change to your files, like a time machine. Created by Linus Torvalds. Install once.
GitHub
A website that stores Git-tracked projects in the cloud. Free for most uses. Think Google Drive but for code. Git is the tool. GitHub is the website.
Repository (Repo)
A single project on GitHub. Just a folder of files with version history.
API
Application Programming Interface. A way for one program to talk to another. "Calling the OpenAI API" means sending a request over the internet to OpenAI's servers and getting an answer back.
API key
A long secret password that proves you're allowed to use an API. Treat it like a credit card.
Frontend / Backend
Frontend = visible part (browser/phone). Backend = invisible server part. Database = storage.
Framework
Pre-made skeleton of code that handles common patterns. React for frontends, Next.js wraps React. AI will pick for you.
Library / Package
Reusable chunk of code someone else wrote that you "import." react, lodash, tailwind.
Supabase
Free/cheap service that gives you a database, logins, file storage with almost no setup. Lovable, Bolt, Antigravity all connect to it.
Token (AI)
A chunk of text, roughly ¾ of a word. AI tools charge by tokens. "5M tokens" ≈ 5 million words of conversation.
Context window
How much text an AI holds in its head at once. Bigger = it can read more of your code before answering.
Prompt
The instruction you give the AI. "Build a to-do app with dark mode" is a prompt.
Agent
An AI that takes actions, not just answers questions. Reads files, runs commands, browses the web, edits code on its own. Antigravity, Claude Code, Cowork, Codex are agentic.
MCP
Model Context Protocol. An open standard, started by Anthropic, for hooking AI tools into external services like Slack, Notion, Gmail, Supabase.
Artifact
A file the AI produces as proof of its work — a plan, a summary, a test result.
Sandbox / VM
Isolated mini-computer the AI runs commands inside, so it can't trash your real machine.
Deployment / Hosting
Putting your app on a public computer so the world can use it. Common free hosts: Vercel, Netlify, Cloudflare Pages, Render.

Part Two

Coding basics — the mental scaffolding

You won't write much code. But understanding how code thinks makes every AI conversation faster, every bug less mysterious, and every tool more usable.

Why this part exists

Vibe coding isn't no-code. It's less code. When AI writes something, you'll occasionally need to peek at it. When something breaks, you need to be able to describe the problem precisely. These eight chapters give you exactly enough vocabulary to do that — no more, no less.

CHAPTER 2.1 · L1 Toy

How the internet actually works

Every app you'll build uses the same plumbing. When you understand the plumbing — even at a cartoon level — you'll stop being mystified by errors like "404" or "CORS error" or "the API is down."

The basic transaction

Whenever you type a URL into your browser, this happens:

Browser
Sends a request to a server: "give me this page."
→
DNS
Looks up which computer the domain points to.
→
Server
Receives the request, prepares a response.
→
Browser
Receives the response, displays it.

This is HTTP (HyperText Transfer Protocol). Every web request is HTTP. Your browser sends a request; the server sends a response. That's it.

The HTTP verbs you'll see in your AI conversations

VerbMeansExample
GET"Give me data"Loading a webpage, fetching a list of users
POST"Create something new"Submitting a signup form
PUT / PATCH"Update something"Editing your profile
DELETE"Remove something"Deleting a post

Status codes — what the server says back

Every HTTP response includes a 3-digit code. The first digit tells you the family:

  • 2xx — Success. 200 OK is the most common: "here's what you asked for."
  • 3xx — Redirect. "Go look over there instead."
  • 4xx — You messed up. 404 = page not found. 401 = you're not logged in. 403 = you can't access this. 400 = your request was malformed.
  • 5xx — Server messed up. 500 = something broke on the server. 503 = server overloaded.

What's a "server" anyway?

A server is just a computer running 24/7 with a piece of software that listens for requests. It could be a giant Amazon data center or a Raspberry Pi in your closet — same idea. When you "deploy" your app to Vercel or Netlify, you're putting your code on their servers.

The browser developer console

Press F12 in any browser. The Network tab shows every HTTP request your app made — green ones succeeded, red ones failed. When you tell an AI "my app isn't loading data," screenshot this tab. It's gold.

CORS in one sentence

If you ever see a "CORS error," it means: the browser blocked your frontend from talking to a different server because that server didn't explicitly allow it. Tell the AI: "I'm getting a CORS error from [URL]. Add the right CORS headers." It'll handle it.

CHAPTER 2.2 · L1 Toy

Data structures: arrays, objects, JSON

Apps move data around constantly. There are really only three shapes of data you need to recognize on sight.

1. Values

The basics:

// Numbers
const age = 28;
const price = 49.99;

// Strings (text)
const name = "Maya";

// Booleans (true/false)
const isLoggedIn = true;

// Empty / nothing
const nothing = null;

2. Arrays — ordered lists

Square brackets [ ]. Items separated by commas. Indexed from zero, not one.

const fruits = ["apple", "banana", "cherry"];

fruits[0]   // "apple"   ← yes, zero, not one
fruits[1]   // "banana"
fruits.length    // 3

Anywhere you see "a list of things" in an app — list of users, list of habits, list of messages — under the hood, that's an array.

3. Objects — labeled boxes

Curly braces { }. Each piece of data has a key (label) and a value.

const user = {
  name: "Maya",
  age: 28,
  isLoggedIn: true,
  email: "maya@example.com"
};

user.name        // "Maya"
user.email       // "maya@example.com"

Anywhere you see "a thing with several attributes" — a user, a product, a blog post — that's an object.

The combo: array of objects

The most common shape in real apps. A list, where each item has multiple fields:

const habits = [
  { name: "Drink water", icon: "💧", streak: 12 },
  { name: "Read",        icon: "📖", streak: 3  },
  { name: "Walk",        icon: "🚶", streak: 21 }
];

habits[0].streak  // 12
habits.length      // 3

JSON — the universal exchange format

JSON (JavaScript Object Notation) is just text that looks like arrays and objects. It's how data travels across the internet between apps. Same shape as the code above, with quotes around the keys:

{
  "habits": [
    { "name": "Drink water", "streak": 12 },
    { "name": "Read", "streak": 3 }
  ],
  "user": "maya@example.com"
}

Every API response you'll see in this guide is JSON. When you call OpenAI's API, the answer comes back as JSON. When your frontend talks to your backend, it's JSON. Learn to read it; you'll read it daily.

A useful habit

When debugging, ask the AI: "Show me the exact JSON your backend is returning for this request." Then you can spot if a field is missing, named wrong, or has the wrong type.

CHAPTER 2.3 · L1 Toy

Logic basics: if, loops, functions

The three building blocks of every program. Once you can read these, you can read 70% of any AI-generated code.

1. if / else — making decisions

"If this is true, do X. Otherwise, do Y."

if (user.age >= 18) {
  showAdultContent();
} else {
  showKidContent();
}

Comparisons: === means "equal to" (use this), !== means "not equal." >, <, >=, <= as you'd expect.

Combine with && (and) and || (or):

if (user.isLoggedIn && user.age >= 18) {
  // must be logged in AND 18+
}

2. Loops — doing something for each item

The most common pattern: "for each thing in this list, do something."

for (const habit of habits) {
  console.log(habit.name);
}

In modern JavaScript and React, you'll mostly see this written as .map() — which builds a new list by transforming each item:

// take each habit and pull out just its name
const names = habits.map(habit => habit.name);
// names = ["Drink water", "Read", "Walk"]

And .filter() — which builds a new list of only the items that match a condition:

const longStreaks = habits.filter(h => h.streak > 10);
// longStreaks = items where streak is greater than 10

3. Functions — reusable bundles of logic

A function is a labeled chunk of code you can call repeatedly. Takes inputs (parameters), returns an output.

function greet(name) {
  return "Hello, " + name + "!";
}

greet("Maya");   // "Hello, Maya!"
greet("Sam");    // "Hello, Sam!"

You'll see functions written three ways in modern JS, all meaning the same thing:

// classic
function add(a, b) { return a + b; }

// arrow function (the cool modern way)
const add = (a, b) => a + b;

// arrow with multi-line body
const add = (a, b) => {
  return a + b;
};

If you can identify a function definition, a loop, and an if statement in a file, you can have a productive conversation with AI about that file.

CHAPTER 2.4 · L2 Personal

How databases work

A database is just persistent storage — data that survives a refresh, a shutdown, a deploy. The most common kind for beginners is a relational database, which thinks in tables.

Tables, rows, columns

Picture a spreadsheet. The whole sheet is a table. Each horizontal line is a row (one record). Each vertical line is a column (one field).

┌─────────── TABLE: habits ───────────────────────┐ │ id │ name │ icon │ user_id │ … │ ├───────┼──────────────┼─────────┼───────────┼─────┤ │ 1 │ Drink water │ 💧 │ abc-123 │ … │ ← row │ 2 │ Read │ 📖 │ abc-123 │ … │ ← row │ 3 │ Meditate │ 🧘 │ xyz-789 │ … │ ← row └───────┴──────────────┴─────────┴───────────┴─────┘ ↑ ↑ column column

Every table has a primary key — usually called id — that uniquely identifies each row. Some tables also have foreign keys — columns that point to a row in another table (here, user_id points to a row in a users table).

The four operations: CRUD

Everything you do to a database is one of four things, abbreviated CRUD:

  • Create — add a new row
  • Read — fetch one or more rows
  • Update — change a row
  • Delete — remove a row

"Build a CRUD app" means "build an app that lets users create, read, update, and delete some kind of record." It's the bread and butter of vibe coding.

SQL — the language databases speak

You'll rarely write SQL by hand (AI will), but recognizing it helps. SQL stands for Structured Query Language.

-- Read all habits for one user
SELECT name, icon FROM habits WHERE user_id = 'abc-123';

-- Create a new habit
INSERT INTO habits (name, icon, user_id)
VALUES ('Stretch', '🤸', 'abc-123');

-- Update a habit's name
UPDATE habits SET name = 'Yoga' WHERE id = 3;

-- Delete a habit
DELETE FROM habits WHERE id = 3;

Supabase — your beginner-friendly database

Supabase is a free service that gives you a real database (PostgreSQL), built-in user authentication, and file storage. You configure tables in a friendly web UI; AI generates the queries.

To create a habits table in Supabase, you'd click "New table" and define columns. AI can do this for you via Supabase's MCP server, or by writing SQL.

Row Level Security — turn this on

By default, anyone who knows your Supabase URL can read your entire database. Row Level Security (RLS) fixes that — it restricts which rows a user can see. Always say in your prompts: "use Supabase with Row Level Security enabled and policies so users only see their own rows." This is the #1 vibe coding security mistake.

NoSQL — the other flavor

Some databases (MongoDB, Firestore) don't use tables — they store JSON-like documents. You may bump into these. The mental model is similar: "collections" instead of tables, "documents" instead of rows. Start with Supabase; come back if you ever need NoSQL.

CHAPTER 2.5 · L2 Personal

Version control: branches, merges, conflicts

You met Git briefly in Part 1. Here's a deeper picture, because Git is the safety net that lets you take risks with AI.

The model: snapshots on a timeline

Every git commit is a snapshot of your entire project at that moment. Git keeps them all, with messages. Imagine a vertical line, with each commit a dot:

main branch ● "Initial commit" │ ● "Add login page" │ ● "Wire up Supabase auth" │ ● "Add dashboard" ← you are here

You can travel back to any commit. The git log command shows them all. In VS Code's Source Control panel you'll see this visually.

Branches — parallel timelines

A branch is a parallel line of commits. The main one is called main (sometimes master). You can create a new branch to try something risky without affecting main.

main ●──●──●──● │ ●──● experiment-dark-mode (branch)

You'd create the branch with git checkout -b experiment-dark-mode. Now any commits go to that branch only. main stays clean.

Merging — bringing branches back together

When the experiment works, you merge it back into main. The branches join up:

main ●──●──●──●──────● ← merge commit │ ╱ ●──●╱ experiment-dark-mode

From your main branch: git merge experiment-dark-mode. Done.

Conflicts — when Git can't auto-merge

If both branches edited the same line of the same file in different ways, Git can't decide. It marks the conflict and asks you to pick:

<<<<<<< HEAD
const theme = "light";
=======
const theme = "dark";
>>>>>>> experiment-dark-mode

You (or AI) edit the file to pick the version you want, then commit again. Conflicts feel scary the first time. After three, they're routine.

Pull requests (PRs)

On GitHub, when you push a branch, you can open a pull request — basically: "hey team, please review my branch before we merge it into main." Solo, you'll PR yourself; with collaborators, this is how all changes flow.

The everyday rhythm

  1. Before a risky AI task: git checkout -b try-something
  2. Let AI work. Commit as you go: git add . && git commit -m "wip"
  3. If it works: git checkout main && git merge try-something
  4. If it doesn't: git checkout main — you're back to safety. Delete the branch with git branch -D try-something.
The killer phrase

Tell your AI tool: "Before you make these changes, create a new Git branch called X. Work on it. I'll merge to main only if I like the result." Antigravity, Cursor, Claude Code all handle this.

CHAPTER 2.6 · L3 Shared

Authentication & sessions

Almost every app you'll build needs logins. The mechanics are simpler than they look once you see the flow.

The login dance

1. User
Enters email + password.
→
2. Server
Checks against stored hash. If match: issues a token.
→
3. Browser
Stores token. Sends it with every future request.
→
4. Server
Verifies token on each request. "Yep, this is Maya."

Why passwords aren't stored as-is

If a server stored "maya@example.com / hunter2" in plain text and got hacked, every account is exposed. Instead, servers store a hash — a one-way scramble. "hunter2" hashes to something like $2b$10$N9qo8uLOickgx2ZMRZoMye…. Even the server can't reverse it. To check a login, they hash what the user typed and compare hashes.

You will never write password-hashing code yourself. Supabase Auth, Auth0, Clerk, Firebase Auth all handle it. But you should ask, when AI builds auth for you: "are you storing passwords as bcrypt hashes, not plaintext?"

Sessions and tokens

Two main schemes you'll see:

  • Session cookies. Server stores session info, browser holds an opaque ID in a cookie. Older but solid.
  • JWT (JSON Web Token). A self-contained encrypted token. Browser holds it; server verifies it on each request. Modern apps lean this way.

Supabase uses JWTs. The token includes the user's ID, email, and expiration time, signed by Supabase so it can't be forged. When the frontend makes a request, it attaches the token in a header: Authorization: Bearer eyJhbGc...

OAuth — "sign in with Google"

OAuth is the protocol behind every "Sign in with [Google / GitHub / Apple]" button. The dance:

  1. User clicks "Sign in with Google."
  2. They're sent to Google's login page.
  3. They confirm "yes, let this app know my email."
  4. Google redirects them back to your app with a token.
  5. Your app uses the token to know who they are.

You never see their Google password. Supabase + Lovable + Bolt all support OAuth out of the box.

Authorization vs Authentication

  • Authentication = "who are you?" (login)
  • Authorization = "what are you allowed to do?" (permissions)

Row Level Security in Supabase is authorization. Authentication says "this is Maya"; authorization says "Maya can only see her own habits, not Sam's."

The mistake everyone makes

Building auth that "works" but doesn't check authorization. A user can be logged in (authenticated) and still hit URLs to view other people's data because the server forgot to check. After AI builds login, always ask: "now write authorization checks that make sure users can only see/edit their own data. Show me where you added them."

CHAPTER 2.7 · L3 Shared

How APIs actually work end-to-end

You'll call APIs constantly: the OpenAI API for AI features, the Stripe API for payments, the Supabase API for data. They all follow the same pattern.

The anatomy of an API call

Here's a typical request to OpenAI, written in JavaScript:

const response = await fetch("https://api.openai.com/v1/chat/completions", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": "Bearer " + apiKey
  },
  body: JSON.stringify({
    model: "gpt-5.3",
    messages: [{ role: "user", content: "Write a haiku about cats" }]
  })
});

const data = await response.json();

Five things to notice:

  1. URL — where the request goes. APIs have many endpoints (URLs) for different actions.
  2. Method — POST here, because we're creating a new completion. (Recall HTTP verbs from 2.1.)
  3. Headers — metadata. Authorization proves who you are. Content-Type says "I'm sending JSON."
  4. Body — the actual data. JSON describing what you want.
  5. Response — also JSON. You "await" because it travels over the network and takes time.

The big risk: API keys

That apiKey variable above? It's a secret that proves you're you, attached to a credit card. It must never appear in your frontend code, because anyone using your app could open DevTools and steal it. Two safe patterns:

  • Environment variables — store the key in a .env file (never committed to Git) and read it from there server-side.
  • Backend proxy — your frontend calls your backend, your backend calls OpenAI with the key, returns the result. The key never leaves your server.

Lovable, Bolt, Replit have a "Secrets" or "Environment Variables" section. Put your API keys there. AI will know how to use them.

Rate limits — every API has them

APIs cap how many requests you can make per minute/hour. If you exceed it, you get 429 Too Many Requests. Your app should handle this — usually by waiting and retrying. Ask AI: "add exponential backoff retries for 429 errors."

SDKs — easier wrappers

Most big APIs (OpenAI, Anthropic, Stripe, Supabase) provide official SDKs — pre-written code that hides the fetch/headers/JSON detail. The OpenAI call above, using the SDK, becomes:

const result = await openai.chat.completions.create({
  model: "gpt-5.3",
  messages: [{ role: "user", content: "Write a haiku about cats" }]
});

Cleaner, but same idea underneath. AI defaults to SDKs when they exist.

Webhooks — the reverse direction

Normally your app calls an API. A webhook is the reverse: an outside service calls your app when something happens. Examples: Stripe webhooks tell your app when a payment succeeds; GitHub webhooks tell your app when someone pushes code.

You'll set webhooks up by giving the outside service a URL on your server, like https://myapp.com/webhooks/stripe. Modern hosts (Vercel, Render) handle this automatically.

CHAPTER 2.8 · L3 Shared

Environments & env variables

An "environment" is a copy of your app running somewhere. Real apps usually have three:

EnvironmentWhat it isWho uses it
DevelopmentRunning on your laptop while you buildYou
StagingLive on the internet but pre-releaseTeam / testers
ProductionThe real app real users useThe public

You don't need a staging environment for your first apps. But the concept matters because each environment uses different settings — different API keys, different databases, different URLs. That's where env variables come in.

Environment variables (env vars)

Variables that exist outside your code, configured per environment. Your code reads them without knowing what's in them. Example:

// in your code (same for all environments)
const apiKey = process.env.OPENAI_API_KEY;
const dbUrl = process.env.DATABASE_URL;

In development, those come from a .env file on your laptop:

OPENAI_API_KEY=sk-test-1234...
DATABASE_URL=postgresql://localhost/myapp_dev

In production, they come from your hosting dashboard (Vercel, Render, etc.):

OPENAI_API_KEY=sk-prod-real-key...
DATABASE_URL=postgresql://prod-server.amazonaws.com/myapp

The .gitignore file — critical

You never commit .env to Git. It contains real secrets. Every project must have a .gitignore file listing what Git should skip:

# .gitignore
.env
.env.local
node_modules/
.DS_Store

AI tools create this for you. But always check that .env is listed before pushing to GitHub for the first time.

If you do leak a key

Bots scan public GitHub repos for API keys within seconds. If you accidentally push one, don't just delete the commit — immediately revoke the key in the provider's dashboard and generate a new one. Deleting the commit isn't enough; the leaked version exists in Git history forever and on attackers' computers.

Beyond keys: feature flags

Env vars also let you toggle features per environment. E.g. ENABLE_NEW_PRICING=true in staging, false in production. Your code: if (process.env.ENABLE_NEW_PRICING === "true") { ... }. You roll out features safely.

You don't need this on day one. But by Level 4 (SaaS) you will, and now you know the term.

CHAPTER 3.1 · L1 Toy

Installing your starter stack

Total time: ~30 minutes. After today these are background scenery.

Install Node.js

Go to nodejs.org, download the "LTS" version (the stable one). Run the installer. Click through. Leave "automatically install necessary tools" checked. This installs Node, npm, and npx all at once.

The cardinal rule: small steps, frequent commits

Biggest beginner mistake: too much at once. "Build a complete Instagram clone with stories, DMs, and reels" produces a janky mess that's hard to fix. "Build a single page where users sign up and log in" works beautifully.

After it works, commit. Next slice: "Now a page where logged-in users upload a photo and see all their uploads." Commit. Repeat.

Ten small commits beat one giant generation every time. When something breaks, you can roll back five minutes — not five days. The rule every vibe coder learns the hard way

The 80/20 of avoiding agent disasters

  1. Commit before any big ask. Always.
  2. Stay in "Ask permissions" mode in Claude Code, Cursor, Antigravity. Watch the diffs. You'll learn what good code looks like by osmosis.
  3. If the AI is in a doom loop (same fix five times, getting worse), stop it. Roll back. Start fresh with a tighter prompt.
  4. When debugging, share the exact error, not your interpretation.
  5. Periodically ask AI to summarize what it's done. Forces a re-read; catches drift.
CHAPTER 5.2

Prompting like a pro

Your prompts are the source code of vibe coding. A great prompt turns a 30-minute mess into a 30-second build.

The four-part prompt

Context: "I'm building a habit tracker for myself."
Goal: "Right now I need sign-up and login."
Constraints: "Supabase auth. Email + password. Tailwind. Minimal — one column, centered, no animations."
Acceptance: "Done when I can sign up, log out, log back in, session persists through refresh."

That prompt produces something usable on first try about 80% of the time. Compare to "make a login page lol" — which leaves ten things to guess.

Eleven prompting tricks that punch above their weight

  • Show, don't just tell. "Make it look like Linear.app's homepage" beats "make it modern."
  • Specify the stack. "Use React, Tailwind, Supabase, no other libraries."
  • Define the data shape. "Habits have a name, icon, color, and a daily log of timestamps."
  • State what NOT to do. "Don't add comments. Don't write tests yet. Don't touch other pages."
  • Ask for a plan first. "Before writing code, describe in 3-5 bullets what you'll do and which files you'll touch."
  • Reference existing code. "Mimic the pattern from the Habits page when building Goals."
  • Tell it what good looks like. "Mobile-first. WCAG-AA contrast. Loads in under 2 seconds."
  • Use Markdown structure. Bullets and headings in your prompt make AI follow instructions more reliably than walls of text.
  • Be explicit about edge cases. "What if the user has no habits yet? Show empty state with a button to add the first."
  • Ask for verification. "After you're done, walk through the user flow step by step." Antigravity and Codex will literally drive their browser to do this.
  • End with success criteria. "Don't mark complete until X, Y, Z all pass."

Three back-pocket prompts

When stuck

"Stop what you're doing. Read the entire codebase. Describe in plain English what this app does and how the pieces fit. Then propose three different approaches to solve [problem]. Don't write code yet."

When AI keeps making the same mistake

"You've tried [X] three times and it keeps failing because [Y]. Forget that approach entirely. What's a completely different way to achieve the goal?"

Add a feature without breaking anything

"I want to add [feature]. First, list every file you'll need to change and why. Show me the list. Don't change anything until I approve. After I approve, make changes one file at a time and pause between files."

Build a single-file HTML pomodoro timer. Visible clock showing minutes:seconds. Three buttons: Start, Pause, Reset. A toggle for 25/5 (focus/break) vs 50/10 vs custom. Big readable type on dark background. Audio chime when a session ends (built into HTML, no external file). Everything in one .html file with inline CSS and JavaScript. No frameworks.

What you'll learn at L1

  • How a single HTML file actually works (the spec → generate → run loop).
  • How to read JavaScript well enough to spot obvious things (where the timer logic lives).
  • How to iterate: ask AI to change one thing, see what happens.
  • How to commit to Git for the first time (use VS Code's Source Control panel — it's a one-click experience).

Five more toys to try

  1. A countdown to a date you care about (your birthday, a vacation).
  2. A "decision wheel" that randomly picks an option from a list you type in.
  3. A unit converter (miles ↔ km, F ↔ C).
  4. A "rate my plant" image-upload that uses a fake AI score.
  5. A typing speed test.

You'll feel almost embarrassingly capable by toy five. That's the point.

CHAPTER 6.3 · L2 Personal

Level 2 — personal tools

Now your app saves data. Just for you. No accounts — but the data sticks around between visits.

What's new at L2

  • Persistence. Either browser localStorage (data lives in the user's browser only) or a real database.
  • Multiple pages or screens. A nav. A real layout.
  • Hosting. Lovable's publish, or Vercel/Netlify if you went the IDE route.
  • Mobile-responsive. Use it from your phone.

Concrete project: personal daily journal

L2 project prompt

Build "Pages" — a personal journal app for one user (me). No authentication. Use Supabase for storage but skip auth — just a single hardcoded user. Two pages: Today (a big textarea for today's entry, autosaves every 5 seconds) and History (a list of past entries by date, click to view full entry). Show a calendar heatmap on History (like GitHub contributions) of which days I journaled. Tailwind, serif font for the writing area, sans-serif everywhere else. Beautiful, minimal, calm.

What you'll learn at L2

  • How a real database works (Part 2.4).
  • How to deploy to the public internet.
  • How to read your app from your phone.
  • The first taste of "oh, I made a real product."

Five more personal tools

  1. A reading log: books you've read, notes per book.
  2. A workout tracker.
  3. A flashcard study app built from your notes.
  4. A meal planner with grocery-list export.
  5. A bookmark organizer with tags and full-text search.
When L2 secretly becomes L3

You build a personal tool. A friend says "can you make me one?" Don't duplicate the project. Add real auth, give them an account. Congratulations — you've graduated to L3.

CHAPTER 6.4 · L3 Shared

Level 3 — shared apps

Now other people use it. Real accounts. Real privacy. Free, but the consequences of bugs are real.

What's new at L3

  • Authentication (Part 2.6). Email/password or OAuth.
  • Authorization — Row Level Security so users only see their own data.
  • Error handling that doesn't crash on weird input.
  • Empty states, loading states, failure states.
  • A landing page explaining what your app does.
  • A privacy policy (use a free generator).

Concrete project: book club tracker

L3 project prompt

Build "Pageturners" — a free book club tracker. Users sign up with email or Google OAuth via Supabase, with Row Level Security so each user only sees clubs they're a member of. Features: (1) Create a club, get an invite link. (2) Members vote on the next book from a shared list. (3) Track who's read it (a checkbox). (4) Discussion thread per book with markdown comments. (5) "Next meeting" date with email reminders 24h before. Use React, Tailwind, Supabase. Include a public landing page at "/" explaining the app, plus /privacy and /terms boilerplate pages. Handle all common error cases gracefully — show toasts for errors, never just crash.

What you'll learn at L3

  • How auth actually flows end to end.
  • How to test something with two different user accounts (most bugs at this level are "I forgot the other user perspective").
  • How to write a clear landing page that explains what your thing does.
  • What "production-quality polish" actually feels like — error states, loading spinners, empty states.
  • How a privacy policy looks.
The L3 trap

Shipping an L3 app with L2 security. The classic mistake: auth works, but you forgot to enable Row Level Security on a table, so anyone with the URL can see everyone's data. Always test by signing up as User B and trying to access User A's URLs directly. If it works — you have a leak. Ask AI: "audit this app for missing authorization checks; show me where users could access other users' data."

CHAPTER 6.5 · L4 SaaS

Level 4 — SaaS / paid products

People pay. The skills change. You're not just a builder; you're operating a business.

What's new at L4

  • Payments. Stripe Checkout for one-time, Stripe Subscriptions for recurring.
  • Webhooks (Part 2.7) for handling Stripe events — successful charges, failed renewals, refunds.
  • Tiered access. "Free users get X. Paid users get Y."
  • Email infrastructure. Welcome emails, password resets, billing receipts. Use Resend or Loops.
  • Customer support. Even a "support@yourapp.com" inbox is a start.
  • Analytics. PostHog or Plausible. Know what people actually do.
  • A real domain. Custom domain ($10/year), not .lovable.app.
  • Backups. Supabase has them automatically; verify they exist.

Concrete project: a writing assistant SaaS

L4 project prompt (build in stages)

Build "Sharpen" — a $12/month AI writing assistant. Users paste a draft; the app gives back specific suggestions (cut weak verbs, suggest stronger nouns, flag passive voice, suggest cuts). Free tier: 3 documents per month. Paid tier: unlimited. Stack: React, Tailwind, Supabase auth + DB, Stripe Subscriptions, OpenAI API on the backend (key never exposed to frontend). Include: landing page with pricing table, free signup flow, paywall when a free user hits the limit, billing page where users can cancel, automatic free downgrade if their subscription lapses, transactional emails on signup and cancellation. RLS on every table. Include /privacy, /terms, /refund-policy pages. Build it in three milestones: (1) free tier only with paywall message; (2) wire up Stripe; (3) email + analytics.

What you'll learn at L4

  • Stripe end-to-end — checkout, webhooks, customer portal.
  • How to handle the edge cases that make customers angry (failed payment, double-charge, refund flow).
  • How to build the "non-product" pages users expect (pricing, terms, refunds).
  • What "customer support" actually feels like (it's mostly password resets at this scale).
  • How to read product analytics and decide what to build next.
The honest L4 reality

The technical part is the easy half. The hard half is marketing, sales, customer feedback loops, content, and the slog of getting from $0 to $1,000 MRR. Vibe coding gives you a product. Getting paid for it is its own skill — orthogonal to anything in this guide. Indie Hackers and the "build in public" community on X are where to learn that piece.

CHAPTER 6.6 · L5 Production

Level 5 — production-grade

Hundreds of thousands of users. Millions. You're operating software at scale. Vibe coding gets you here, but the skills you need become different — less about building, more about running.

What's new at L5

  • Monitoring. Sentry for errors. Datadog or Better Stack for uptime. Slack alerts when something breaks.
  • Multiple environments. dev, staging, production (Part 2.8).
  • CI/CD. Tests run automatically on every push. Failed tests block merging.
  • Performance. Database indexes, caching (Redis), CDNs, optimizing slow queries.
  • Compliance. GDPR (data deletion requests). SOC 2 if you sell to bigger companies. Cookie consent banners. Terms with real lawyers.
  • Security audits. Penetration testing. Bug bounties.
  • On-call. Someone (probably you) gets paged at 2am when servers crash.
  • A team. Solo founders rarely operate at L5 alone. You hire, or you contract.

Concrete project: production hardening

At L5, projects aren't "build a thing" — they're "make this thing reliable, fast, secure, and observable." Here's what a real L5 project sprint looks like:

L5 hardening sprint

Take "Sharpen" from Level 4 to production-grade. Tasks: (1) Add Sentry for error tracking, alert to Slack on every 500 error. (2) Set up a staging environment that mirrors production. (3) Write a CI pipeline that runs tests on every PR — block merges if tests fail. (4) Add database indexes on the most-queried columns; you'll need to identify them first by analyzing slow queries. (5) Add Redis caching for the OpenAI API responses (same input = cached output, expires daily). (6) Build a GDPR data-deletion endpoint that wipes a user's data from all tables. (7) Add a cookie consent banner. (8) Get a real privacy policy from Termly. (9) Write a runbook in /docs/runbook.md for the most common production issues. Do this in nine separate PRs, each branched, reviewed, and merged independently.

What you'll learn at L5

  • How to read code well enough to spot when AI is wrong about something subtle.
  • How to think about systems, not just features. (Caching, queues, retry policies.)
  • How to write a runbook a stranger could follow at 2am.
  • How to triage incidents calmly.
  • How to say "no" to features that would compromise reliability.

The honest truth about L5 and AI

AI tools are great at L1–L4. They're useful but not sufficient at L5. The skills that matter most at L5 — judgment under pressure, deep debugging, security thinking, system design — are exactly the skills AI is least reliable at. People who reach L5 are people who used the L1–L4 years to actually learn how code works, not just how to prompt for it.

L5 isn't a different version of vibe coding. It's a different job. You've stopped being a builder and become an operator. The level where this guide hands you off

CHAPTER 7.2 · L2–L5

Debugging when things break

Things will break. The frequency drops a lot once you internalize "small steps, frequent commits" — but never to zero. The order of operations:

  1. Reproduce it. Make it happen on purpose. Note the exact steps.
  2. Capture the error. If your browser console (F12 → Console) shows red, copy the whole thing.
  3. Paste it into your AI verbatim: "This error appears when I click [X]. The full error: [paste]. The code where this happens is in [file]. Please investigate before changing anything."
  4. If it doesn't fix in two attempts, roll back to your last working commit (git reset --hard HEAD~1 or VS Code Source Control panel).
  5. Try a different approach. Start a fresh AI session. Don't carry broken context.

The browser DevTools every vibe coder should know

Press F12 in any browser:

  • Elements — inspect any HTML element and see its styles. Right-click → Inspect.
  • Console — see error messages and logs. Most JS bugs surface here.
  • Network — every request your app makes. Red = failed.
  • Application — what's in cookies and localStorage. "Why isn't my user staying logged in?" bugs live here.

When debugging with AI: "Here's what the console shows: [paste]. Here's the failed request from the Network tab: [paste]." Specificity makes everything 10× faster.

CHAPTER 7.4

Pitfalls to avoid

Read twice.

1. API keys in a public GitHub repo

Push code with an API key inside → scrapers find it within minutes → $4,000 bill in the morning. Always env variables. Always .env in .gitignore. If leaked, immediately revoke in the provider's dashboard — don't just delete the commit.

2. Skipping Supabase Row Level Security

Default: anyone knowing your Supabase URL can read your whole database. RLS is the fix; must be turned on per-table. Every Supabase prompt should include "with RLS enabled and policies that only let users read their own rows."

3. Never committing to Git

#1 cause of vibe coding tragedies. AI breaks something subtle. Can't undo. Hours lost. Commit at least every 30 min when actively building.

4. Asking for too much in one prompt

"Build me Twitter" isn't a prompt. "Build a sign-up page with Supabase email auth, redirect to a dashboard" is. Slice work into 2-5 min pieces.

5. Trusting AI on security

AI happily builds things that work while being trivially hackable. After anything with logins, payments, user data: "Audit this code for common security issues — XSS, SQL injection, missing auth checks, exposed secrets, weak validation. List every finding before fixing anything."

6. Believing "I tested this and it works"

Test yourself. Click the buttons. Try edge cases. No internet, empty database, invalid input, on a phone. AI tested its mental model — you test reality.

7. Tool-hopping instead of finishing

Every week there's a hot new tool. Pick one workflow and ship something. Switch tools between projects, not mid-build.
